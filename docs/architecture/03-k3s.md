# Arquitetura Interna do K3s

**Projeto:** Plataforma Digitalead
**Documento:** Arquitetura 03
**Versão:** 1.0
**Status:** Aprovado
**Última atualização:** 2026-07-13

## 1. Objetivo

Este documento explica como o K3s funciona internamente, quais componentes ele fornece e por que foi escolhido como distribuição Kubernetes da Plataforma Digitalead.

Este documento não contém o procedimento de instalação.

A instalação será documentada em:

```text
docs/bootstrap/03-instalacao-k3s/README.md
```

## 2. Contexto

A Plataforma Digitalead precisa executar aplicações em Kubernetes, mas ainda está na fase inicial da startup.

A arquitetura precisa equilibrar:

* baixo custo;
* compatibilidade com Kubernetes;
* simplicidade operacional;
* capacidade de crescimento;
* possibilidade de migração futura;
* consumo controlado de recursos.

Um cluster Kubernetes tradicional criado com `kubeadm` exigiria maior quantidade de configuração e administração de componentes individuais.

O K3s reduz essa complexidade ao fornecer uma distribuição Kubernetes integrada e empacotada.

## 3. O que é K3s

K3s é uma distribuição Kubernetes compatível e pronta para produção, empacotada principalmente em um único binário.

Ele mantém as APIs e os conceitos centrais do Kubernetes, mas simplifica a instalação e a operação.

Entre as principais características estão:

* binário integrado;
* containerd embarcado;
* suporte a SQLite como datastore padrão de clusters single-server;
* suporte a embedded etcd;
* suporte a bancos externos;
* componentes adicionais empacotados;
* integração com systemd;
* geração e administração de certificados;
* configuração simplificada.

## 4. Motivo da escolha

O K3s foi escolhido porque atende aos requisitos atuais sem impedir evolução futura.

Comparação conceitual:

| Alternativa | Avaliação para a fase atual                                                                           |
| ----------- | ----------------------------------------------------------------------------------------------------- |
| Minikube    | Excelente para laboratório, mas não escolhido como base de produção                                   |
| kubeadm     | Mais flexível, porém exige maior esforço operacional                                                  |
| RKE2        | Boa alternativa para ambientes com requisitos corporativos e hardening mais rigoroso, mas mais pesada |
| MicroK8s    | Viável, porém o time já está construindo a arquitetura em torno do ecossistema K3s                    |
| AKS/EKS/GKE | Reduzem administração do control plane, mas possuem custo superior                                    |
| K3s         | Melhor equilíbrio atual entre custo, simplicidade e compatibilidade                                   |

## 5. Arquitetura inicial

```text
VPS Contabo
dl-platform-01
        │
        ▼
Ubuntu Server 24.04 LTS
        │
        ▼
systemd
        │
        ▼
k3s.service
        │
        ▼
k3s server
        │
        ├── Kubernetes API Server
        ├── Scheduler
        ├── Controller Manager
        ├── Cloud Controller Manager
        ├── Kubelet
        ├── Kube Proxy
        ├── Containerd
        ├── Datastore SQLite
        └── Controllers internos do K3s
                │
                ▼
        Componentes empacotados
                │
                ├── CoreDNS
                ├── Traefik
                ├── Metrics Server
                ├── ServiceLB
                └── Local Path Provisioner
```

No cenário single-node, a mesma VPS executa:

* control plane;
* worker;
* datastore;
* runtime de containers;
* Pods de infraestrutura;
* Pods das aplicações.

## 6. O processo `k3s server`

O serviço principal será iniciado pelo systemd:

```text
systemd
   │
   ▼
k3s.service
   │
   ▼
k3s server
```

O comando `k3s server` inicia o nó com funções de servidor e agent.

O nó server executa os componentes do control plane e também pode executar workloads, a menos que seja configurado para não receber Pods de aplicação.

Em nosso cluster inicial, o único nó deverá executar ambos os papéis.

## 7. Componentes do control plane

### Kubernetes API Server

É a interface central do Kubernetes.

Ferramentas como:

```text
kubectl
Headlamp
Helm
Controllers
Operators
```

comunicam-se com o cluster através da API.

A porta padrão da API é:

```text
6443/TCP
```

Essa porta não deverá ficar aberta indiscriminadamente para a Internet.

### Scheduler

Decide em qual nó cada Pod será executado.

Como inicialmente existirá apenas um nó, todos os Pods elegíveis serão agendados em:

```text
dl-platform-01
```

Mesmo assim, requests, limits, taints, tolerations e prioridades continuarão sendo relevantes.

### Controller Manager

Executa os controladores que reconciliam o estado atual com o estado desejado.

Exemplo:

```text
Deployment deseja 3 réplicas
          │
          ▼
Controller verifica o estado
          │
          ▼
Cria ou remove Pods
          │
          ▼
Estado atual converge para 3 réplicas
```

## 8. Container runtime

O K3s utiliza containerd como runtime padrão.

Não será necessário instalar Docker na VPS.

O fluxo simplificado é:

```text
Deployment
   │
   ▼
Scheduler
   │
   ▼
Kubelet
   │
   ▼
Containerd
   │
   ▼
Container
```

A ausência do Docker reduz dependências e evita manter dois runtimes sem necessidade.

## 9. Datastore

O Kubernetes precisa armazenar o estado do cluster.

Esse estado inclui:

* Deployments;
* Services;
* ConfigMaps;
* Secrets;
* namespaces;
* RBAC;
* metadados de Pods;
* estado dos controllers;
* configurações do cluster.

### Arquitetura inicial

O cluster utilizará SQLite embarcado.

```text
Kubernetes API
      │
      ▼
Kine
      │
      ▼
SQLite
```

O uso de SQLite é adequado para a arquitetura inicial com um único server.

Ele não oferece alta disponibilidade. Se a VPS estiver indisponível, o control plane e o datastore também estarão indisponíveis.

### Evolução para alta disponibilidade

Quando o cluster evoluir para três servers, poderá utilizar embedded etcd:

```text
dl-platform-01 ── etcd member
dl-platform-02 ── etcd member
dl-platform-03 ── etcd member
```

Com três membros, o cluster tolera a perda de um membro e ainda mantém quorum.

O K3s também suporta datastore externo, mas não será adotado nesta primeira fase.

## 10. Componentes empacotados

O K3s distribui componentes adicionais como AddOns administrados por manifests internos.

### CoreDNS

Responsável pela resolução de nomes dentro do cluster.

Exemplo:

```text
backend.dl-prod.svc.cluster.local
```

Uma aplicação normalmente poderá acessar um Service no mesmo namespace utilizando apenas:

```text
backend
```

### Traefik

Será o Ingress Controller inicial.

Fluxo externo:

```text
Internet
   │
   ▼
217.216.55.208:443
   │
   ▼
Traefik
   │
   ▼
Ingress
   │
   ▼
Service
   │
   ▼
Pod
```

Não será instalado NGINX Ingress Controller nesta fase.

### Metrics Server

Fornece métricas básicas de CPU e memória utilizadas por:

```text
kubectl top nodes
kubectl top pods
HorizontalPodAutoscaler
Headlamp
```

Metrics Server não substitui Prometheus. Ele fornece métricas operacionais de curto prazo para o Kubernetes.

### ServiceLB

Implementação leve de Services do tipo `LoadBalancer`.

Em nosso single-node, ele permite que o Traefik utilize as portas públicas do host.

Como o servidor será dedicado ao K3s, as portas `80` e `443` poderão ser utilizadas pelo Ingress Controller.

### Local Path Provisioner

Cria volumes persistentes utilizando o disco local da VPS.

É adequado para a primeira fase, mas possui uma limitação importante:

> O dado permanece associado ao disco do nó.

Em um futuro cluster multi-node, essa estratégia deverá ser reavaliada.

## 11. Diretórios importantes

### Configuração

```text
/etc/rancher/k3s/
```

Arquivo principal planejado:

```text
/etc/rancher/k3s/config.yaml
```

Esse arquivo registrará de forma declarativa as opções utilizadas na inicialização do K3s.

### Kubeconfig

```text
/etc/rancher/k3s/k3s.yaml
```

Esse arquivo concede acesso administrativo ao cluster e não deverá ser armazenado no Git.

### Dados do K3s

```text
/var/lib/rancher/k3s/
```

Esse diretório contém dados importantes da distribuição, incluindo:

* datastore;
* certificados;
* credenciais internas;
* dados dos componentes;
* arquivos utilizados pelo containerd;
* estado do cluster.

### Manifests automáticos

```text
/var/lib/rancher/k3s/server/manifests/
```

Arquivos colocados nesse diretório podem ser aplicados automaticamente ao cluster pelo mecanismo de AddOns do K3s.

Não utilizaremos esse diretório como local principal de nossos manifests de aplicação. Nossa fonte de verdade será o Git.

### Serviço systemd

```text
/etc/systemd/system/k3s.service
```

O arquivo exato poderá variar conforme a instalação, mas o K3s será administrado como um serviço do sistema operacional.

## 12. Fluxo de inicialização

Quando a VPS for ligada:

```text
1. Firmware e kernel iniciam
          │
2. Ubuntu inicializa
          │
3. systemd inicia serviços
          │
4. k3s.service inicia
          │
5. k3s server carrega a configuração
          │
6. datastore é aberto
          │
7. API Server é iniciado
          │
8. Scheduler e controllers são iniciados
          │
9. Kubelet registra o nó
          │
10. Containerd inicia containers
          │
11. CoreDNS, Traefik e componentes internos sobem
          │
12. Node passa para Ready
          │
13. Workloads são reconciliados
```

Esse fluxo será usado como referência para troubleshooting.

## 13. Fluxo de criação de um Pod

Quando um Deployment é aplicado:

```text
Manifest enviado
      │
      ▼
API Server
      │
      ▼
Estado salvo no datastore
      │
      ▼
Deployment Controller
      │
      ▼
ReplicaSet
      │
      ▼
Pod pendente
      │
      ▼
Scheduler seleciona o nó
      │
      ▼
Kubelet recebe a atribuição
      │
      ▼
Containerd baixa a imagem
      │
      ▼
Container é iniciado
      │
      ▼
Readiness Probe aprova
      │
      ▼
Pod fica Ready
```

O K3s não muda esse funcionamento central do Kubernetes. Ele simplifica o empacotamento e a operação dos componentes.

## 14. Certificados

O K3s cria e administra certificados internos utilizados pela comunicação entre os componentes do cluster.

Os certificados internos do Kubernetes são diferentes dos certificados HTTPS públicos utilizados pelos domínios:

```text
app.digitallead.com.br
qa.app.digitallead.com.br
dev.app.digitallead.com.br
```

Os certificados públicos serão posteriormente administrados pelo cert-manager.

## 16. Segurança do datastore

O cluster deverá ser iniciado com criptografia de Secrets em repouso habilitada.

Essa configuração protege o conteúdo dos Secrets armazenados no datastore contra leitura direta do banco sem acesso às chaves de criptografia.

A decisão exata será registrada no arquivo:

```text
/etc/rancher/k3s/config.yaml
```

O backup do diretório e das chaves correspondentes será essencial para recuperação.

## 17. Vantagens da arquitetura escolhida

* baixo consumo operacional;
* Kubernetes compatível;
* instalação simplificada;
* runtime integrado;
* Ingress já disponível;
* métricas básicas já disponíveis;
* adequado para uma única VPS;
* evolução possível para embedded etcd;
* reduz quantidade de componentes administrados separadamente.

## 18. Limitações aceitas

### Ponto único de falha

A única VPS concentra:

* control plane;
* datastore;
* aplicações;
* volumes;
* entrada HTTP e HTTPS.

Uma falha da VPS interromperá toda a plataforma.

### Ausência de manutenção sem indisponibilidade

Reinicializações, upgrades do kernel ou falhas do host podem causar interrupção.

### Storage local

Volumes persistentes dependem do disco da VPS.

### Competição entre ambientes

Produção, QA e desenvolvimento compartilham CPU, memória e disco.

Essas limitações serão mitigadas com:

* backups externos;
* ResourceQuotas;
* Limits;
* monitoramento;
* documentação;
* scripts de recuperação;
* futura separação da produção.

## 19. Quando reavaliar o single-node

A arquitetura deverá ser revisada quando ocorrer um ou mais dos seguintes eventos:

* indisponibilidade deixar de ser aceitável;
* entrada de clientes com SLA;
* manutenção exigir interrupções frequentes;
* uso médio de memória permanecer elevado;
* disco se aproximar do limite definido;
* banco de dados tornar-se altamente crítico;
* aumento significativo do número de usuários;
* necessidade de isolamento físico de produção;
* requisitos regulatórios;
* necessidade de distribuir workloads.

A Plataforma Digitalead adotará inicialmente:

```text
K3s single-node
Datastore SQLite
Containerd embarcado
Traefik habilitado
ServiceLB habilitado
CoreDNS habilitado
Metrics Server habilitado
Local Path Provisioner habilitado
Secrets encryption habilitada
Configuração por /etc/rancher/k3s/config.yaml
```

Essa decisão será revisada conforme a criticidade e a utilização da plataforma aumentarem.
